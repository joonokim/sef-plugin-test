# 검증루프 (Verification Loop) 상세

## 개요

팀 개발에서 각 에이전트의 산출물 품질을 보장하기 위한 반복 검증 프로세스입니다. team-lead (또는 reviewer)가 수행하며, 최대 3라운드까지 진행합니다.

---

## 4개 평가 기준 상세

### 1. 목표 달성 (Goal Achievement)

- [ ] Task에 명시된 요구사항을 모두 충족하는가
- [ ] API 엔드포인트가 설계 명세와 일치하는가
- [ ] 화면 명세서의 입출력 항목이 구현되었는가
- [ ] ROADMAP.md의 해당 Task 완료 조건을 만족하는가

### 2. 완전성 (Completeness)

- [ ] 필요한 파일이 모두 생성되었는가 (controller, service, mapper, dto 등)
- [ ] import/의존성이 누락 없이 선언되었는가
- [ ] 설정 파일 (application.yml, nuxt.config.ts 등)이 업데이트되었는가
- [ ] 에러 처리 코드가 포함되었는가

### 3. 정확성 (Correctness)

- [ ] 컴파일/빌드 에러가 없는가
- [ ] 타입 에러가 없는가 (TypeScript strict mode 기준)
- [ ] 비즈니스 로직이 요구사항과 일치하는가
- [ ] SQL 쿼리/MyBatis 매핑이 정확한가

### 4. 일관성 (Consistency)

- [ ] 기존 코드의 네이밍 컨벤션을 따르는가
- [ ] 프로젝트 아키텍처 패턴 (레이어링, 모듈 구조)을 준수하는가
- [ ] API 응답 형식이 `ApiResponse` 표준을 따르는가
- [ ] 공공 프로젝트: GET/POST only 정책을 준수하는가

## 전략 A/B/C 선택 기준

### 전략 A: 부분 수정

**조건**: 경미한 문제만 존재

- 오타, 누락된 import
- 사소한 네이밍 불일치
- 주석/로그 메시지 수정
- 누락된 null 체크 추가

**실행**: 해당 에이전트에 수정 목록 전달 → 즉시 수정 → 재검증 불필요

### 전략 B: 재작업

**조건**: 구조적 문제 발견

- 잘못된 디자인 패턴 적용
- API 설계 명세와 불일치하는 엔드포인트 구조
- 레이어 간 책임 혼재 (controller에서 비즈니스 로직 수행 등)
- 보안 취약점 (SQL Injection, XSS 등)

**실행**: Task를 반려 → 문제 원인과 올바른 접근 방식 설명 → 에이전트가 처음부터 재구현

### 전략 C: Task 분할

**조건**: Task 범위가 과대하여 단일 에이전트가 처리하기 어려움

- 하나의 Task에 3개 이상의 독립 기능 포함
- BE/FE 양쪽 모두 대규모 변경 필요
- 여러 모듈에 걸친 횡단 관심사

**실행**: 원본 Task를 2~3개 하위 Task로 분할 → `TaskCreate`로 등록 → 적절한 에이전트에 재할당

## 라운드별 에스컬레이션 흐름

```
Round 1: 초기 검증
  ├── 통과 → Task 완료 (TaskUpdate: completed)
  └── 미통과 → 전략 A/B/C 적용 후 Round 2

Round 2: 재검증
  ├── 통과 → Task 완료
  └── 미통과 → 전략 재적용 후 Round 3

Round 3: 최종 검증
  ├── 통과 → Task 완료
  └── 미통과 → AskUserQuestion으로 사용자 판단 요청
              ├── 사용자가 수동 수정
              ├── Task 스킵
              └── 추가 라운드 허용
```

## 에이전트 유휴 상태 모니터링

- 에이전트가 3회 연속 유의미한 진전 없이 유휴 상태 → 교체 검토
- 유휴 판단 기준:
  - `TaskUpdate` 호출 없이 대기
  - 동일한 에러를 반복 발생
  - 산출물 변화량 없음
- 교체 절차:
  1. 해당 에이전트에 `SendMessage` type: `"shutdown_request"` 전송
  2. 새 에이전트 스폰 (동일 역할)
  3. 미완료 Task를 새 에이전트에 재할당
