# 모놀리스 vs 마이크로서비스

## 개요

애플리케이션 아키텍처를 선택할 때 모놀리스와 마이크로서비스 중 어느 것이 적합한지 판단하는 가이드입니다.

## 모놀리스 아키�ecture (Monolithic)

### 정의

모든 기능이 하나의 애플리케이션에 통합된 전통적인 아키텍처입니다.

### 구조

```
┌─────────────────────────────┐
│   Monolithic Application    │
│                             │
│  ┌────────┬──────┬────────┐ │
│  │  User  │Order │Payment │ │
│  │ Module │Module│ Module │ │
│  └────────┴──────┴────────┘ │
│                             │
│  ┌────────────────────────┐ │
│  │   Shared Database      │ │
│  └────────────────────────┘ │
└─────────────────────────────┘
```

### 장점

✅ **개발 단순성**: 하나의 코드베이스, 하나의 배포 단위
✅ **디버깅 용이**: 모든 로그가 한 곳에
✅ **트랜잭션 관리**: ACID 트랜잭션 보장
✅ **성능**: 인메모리 함수 호출 (네트워크 오버헤드 없음)
✅ **배포 간단**: 하나의 애플리케이션만 배포
✅ **작은 팀에 적합**: 초기 개발 속도 빠름

### 단점

❌ **확장성 제한**: 전체 애플리케이션을 스케일해야 함
❌ **배포 위험**: 작은 변경도 전체 재배포 필요
❌ **기술 스택 제약**: 하나의 언어/프레임워크에 종속
❌ **팀 협업 어려움**: 큰 코드베이스에서 충돌 발생
❌ **유지보수 복잡**: 코드가 커질수록 이해하기 어려움

### 적합한 경우

- 작은 프로젝트 (MVP, 스타트업 초기)
- 팀 규모가 작음 (5명 이하)
- 단순한 비즈니스 로직
- 빠른 개발이 필요한 경우
- 트래픽이 예측 가능하고 적음

### 예시

```java
// Spring Boot 모놀리스
@SpringBootApplication
public class MonolithicApp {
    public static void main(String[] args) {
        SpringApplication.run(MonolithicApp.class, args);
    }
}

// 모든 기능이 하나의 애플리케이션에
@RestController
class UserController { ... }

@RestController
class OrderController { ... }

@RestController
class PaymentController { ... }
```

## 마이크로서비스 아키텍처 (Microservices)

### 정의

애플리케이션을 작고 독립적인 서비스들로 분해한 아키텍처입니다.

### 구조

```
                API Gateway
                     │
        ┌────────────┼────────────┐
        │            │            │
   ┌────▼───┐  ┌────▼───┐  ┌────▼────┐
   │  User  │  │ Order  │  │ Payment │
   │Service │  │Service │  │ Service │
   └────┬───┘  └────┬───┘  └────┬────┘
        │           │            │
   ┌────▼───┐  ┌───▼────┐  ┌────▼────┐
   │User DB │  │Order DB│  │Payment  │
   └────────┘  └────────┘  │   DB    │
                            └─────────┘
```

### 장점

✅ **독립적 확장**: 필요한 서비스만 스케일
✅ **독립적 배포**: 다른 서비스에 영향 없이 배포
✅ **기술 자유도**: 서비스마다 다른 기술 스택 사용 가능
✅ **팀 독립성**: 각 팀이 서비스 단위로 작업
✅ **장애 격리**: 한 서비스의 장애가 전체에 영향 최소화
✅ **비즈니스 민첩성**: 빠른 기능 추가/변경

### 단점

❌ **복잡도 증가**: 분산 시스템의 복잡성
❌ **운영 오버헤드**: 여러 서비스 모니터링/배포
❌ **네트워크 지연**: 서비스 간 통신 오버헤드
❌ **데이터 일관성**: 분산 트랜잭션 관리 어려움
❌ **디버깅 어려움**: 여러 서비스에 걸친 문제 추적
❌ **초기 비용**: 인프라 및 도구 설정 필요

### 적합한 경우

- 대규모 프로젝트
- 팀 규모가 큼 (10명 이상)
- 복잡한 비즈니스 도메인
- 서비스별 독립적 확장 필요
- 지속적 배포 및 업데이트
- 높은 가용성 요구

### 예시

```typescript
// User Service (Node.js)
const app = express();
app.get('/api/users/:id', async (req, res) => {
  const user = await userRepo.findById(req.params.id);
  res.json(user);
});

// Order Service (Spring Boot)
@RestController
@RequestMapping("/api/orders")
public class OrderController {
    @PostMapping
    public Order createOrder(@RequestBody OrderRequest req) {
        return orderService.create(req);
    }
}

// Payment Service (Python)
@app.route('/api/payments', methods=['POST'])
def process_payment():
    return payment_service.process(request.json)
```

## 비교표

| 항목 | 모놀리스 | 마이크로서비스 |
|-----|---------|---------------|
| **개발 속도** | 초기 빠름 | 초기 느림 |
| **배포** | 전체 재배포 | 독립 배포 |
| **확장성** | 수평/수직 확장 제한 | 서비스별 확장 |
| **기술 스택** | 단일 스택 | 다양한 스택 |
| **팀 협업** | 코드 충돌 가능 | 팀 독립성 높음 |
| **트랜잭션** | ACID 보장 | Eventual Consistency |
| **모니터링** | 단순 | 복잡 (분산 추적) |
| **인프라 비용** | 낮음 | 높음 |
| **학습 곡선** | 낮음 | 높음 |

## 의사결정 플로우차트

```
프로젝트 시작
    │
    ├─ 팀 크기 < 5명?
    │   └─ Yes → 모놀리스
    │
    ├─ MVP / 프로토타입?
    │   └─ Yes → 모놀리스
    │
    ├─ 트래픽 < 1000 req/min?
    │   └─ Yes → 모놀리스
    │
    ├─ 복잡한 도메인?
    │   └─ Yes → 마이크로서비스 고려
    │
    ├─ 서비스별 독립 확장 필요?
    │   └─ Yes → 마이크로서비스
    │
    └─ 지속적 배포 필요?
        └─ Yes → 마이크로서비스
```

## 하이브리드 접근: 모듈러 모놀리스

모놀리스의 단순성과 마이크로서비스의 구조적 이점을 결합합니다.

```
┌─────────────────────────────────┐
│   Modular Monolithic App        │
│                                 │
│  ┌─────────┐ ┌─────────┐       │
│  │  User   │ │  Order  │       │
│  │ Module  │ │ Module  │       │
│  └────┬────┘ └────┬────┘       │
│       │           │             │
│  ┌────▼───────────▼────┐       │
│  │   Shared Database   │       │
│  └─────────────────────┘       │
└─────────────────────────────────┘
```

**특징**:
- 모듈 간 명확한 경계
- 나중에 마이크로서비스로 전환 용이
- 모놀리스의 단순성 유지

## 공공 프로젝트 권장사항

### 공공 기관 프로젝트

**권장: 모놀리스 또는 모듈러 모놀리스**

이유:
- 단일 WAS 배포 요구사항
- 전자정부프레임워크 표준
- 안정성 및 검증된 기술 선호
- 상대적으로 작은 팀 규모

```yaml
아키텍처: 모놀리스 (Spring Boot WAR)
배포: JEUS/WebLogic
데이터베이스: Oracle (단일 DB)
```

### 대규모 공공 포털

**권장: 하이브리드**

- 공개 포털: 모놀리스 (Nuxt 4 SSR)
- 내부 시스템: 마이크로서비스

## 민간 프로젝트 권장사항

### 스타트업 / MVP

**권장: 모놀리스**

빠른 개발과 검증이 우선순위입니다.

```yaml
아키텍처: 모놀리스
배포: Vercel (프론트) + 단일 백엔드
```

### 성장 단계 (Scaling)

**권장: 모듈러 모놀리스 → 마이크로서비스**

점진적으로 전환합니다.

```
Phase 1: 모듈러 모놀리스
Phase 2: API Gateway 추가
Phase 3: 핵심 서비스 분리 (Auth, Payment)
Phase 4: 완전한 마이크로서비스
```

### 대기업 / 엔터프라이즈

**권장: 마이크로서비스**

```yaml
아키텍처: 마이크로서비스
배포: Kubernetes
인프라: AWS/GCP
```

## 전환 전략 (Migration)

### 모놀리스 → 마이크로서비스

**Strangler Fig Pattern** (점진적 전환)

```
1단계: 모놀리스 앞에 API Gateway 추가
2단계: 새 기능을 마이크로서비스로 개발
3단계: 기존 기능 점진적 분리
4단계: 모놀리스 완전 교체
```

**우선순위**:
1. 변경이 잦은 기능
2. 독립적 확장이 필요한 기능
3. 명확한 도메인 경계가 있는 기능

### 마이크로서비스 → 모놀리스

드물지만 필요한 경우:
- 팀 규모 축소
- 복잡도가 비즈니스 가치를 초과
- 운영 비용 절감 필요

## 결론

**시작은 모놀리스, 필요시 마이크로서비스**

1. **모놀리스로 시작**: 빠른 검증, 낮은 복잡도
2. **모듈화 유지**: 나중에 분리 가능하도록
3. **필요할 때 전환**: 명확한 이유가 있을 때만

> "Don't start with microservices - monoliths are your friend."
> - Martin Fowler
